    Исходящий update заявки (HTTP на стороне 1С)

    POST…/customs-request-updateСервер шлёт изменения после подписи из МП.

    Auth: тот же Bearer, что для создания — из PUT /api/admin/settings/one-c-request-create (oneCRequestCreateBearerToken).

    URL: из oneCRequestCreateUrl в тех же настройках. Путь update:

    • при значении …/customs-request сервер направляет запрос на …/customs-request-update
    • при значении …/customs-request-update URL используется без модификации

    Когда вызывается: после POST /api/customs-requests/:id/files (подпись *_sign из МП). После входящего state от 1С update не отправляется.

    Request (пример, зеркало state):

    {
  "requestId": 123,
  "external1cId": "GUID-REQUEST-1C",
  "status": "in_progress",
  "statusSubType": "primary_documents_sent",
  "statusSubTypeDateTime": "2026-05-27T10:00:00+03:00",
  "dealType": "tripartite",
  "managerExternal1cId": "GUID-MANAGER-1C",
  "managerFullName": "Иванов Иван Иванович",
  "signingFiles": [
    {
      "docType": "contract_sign",
      "fileName": "contract_sign.pdf",
      "mimeType": "application/pdf",
      "fileUrl": "/api/customs-requests/files/uuid.pdf"
    }
  ],
  "source": "import_service_server"
}

    Минимальный набор: external1cId и не менее одного измененного поля. Полный состав полей соответствует POST /api/integration/customs-requests/state (вкладка API для 1С).

    Общие коды для state, МП и исходящего update в 1С: статусы, подстатусы, dealType, docType.

    Справочники (статусы, подстатусы, тип сделки, docType)

    ENUMstatusВерхний уровень заявки (поле state).
    КодВ МПКогда

    RULEdocType + _signДва файла на каждый документ на подпись.

    Для документов на подпись хранятся две версии:

    • Оригинал — docType без суффикса (например contract), приходит из 1С
    • Подписанный — тот же тип + суффикс _sign (например contract_sign), загружает клиент из МП

    Техническое задание (МП ↔ Сервер ↔ 1С)

    СхемаsequenceDiagramОбщий поток (Mermaid). 
    sequenceDiagram
  participant MP as Мобильное приложение
  participant API as Сервер API
  participant C1 as 1С

  MP->>API: Создать заявку + файлы
  API->>API: status new
  API->>C1: requestId + данные + files
  C1-->>API: external1cId
  API->>API: status on_review

  C1->>API: state in_progress + dealType + менеджер
  API->>MP: карточка заявки

  loop Первичный пакет на подпись
    C1->>API: upload → fileUrl
    C1->>API: state signingFiles + subStatus
    API->>MP: скачать оригинал
    MP->>API: upload docType_sign
    API->>C1: подпись видна менеджеру
    Note over C1,MP: при ошибке: statusSubType signature_revision_required
  end

  loop Квитанции и чеки
    C1->>API: квитанции
    MP->>API: чеки оплаты
  end

  loop Изменения в 1С
    C1->>API: state status + statusSubType
    API->>MP: обновление
  end
    ЗаказчикуЧто происходит с заявкой

    Принцип: МП общается только с сервером. 1С общается только с сервером.

    Этапы (по смыслу документа):

    • 1. Создание заявки — МП → сервер → 1С. У заявки стартовый статус «Новая».
    • 2. Обработка в 1С — выбор менеджера и подготовка документов.
    • 3. Первичный пакет на подпись — 1С → МП, затем подпись и возврат в 1С через сервер.
    • 4. Квитанция утилизационного сбора (параллельно п.3) — 1С → МП, затем загрузка чека.
    • 5. Квитанция госпошлины (параллельно п.3) — 1С → МП, затем загрузка чека.
    • 6. Архив перед транзитом — 1С → МП (только скачивание).
    • 7. Итоговые документы после таможни — 1С → МП (только скачивание).

    Статусы (отображение в МП):

    • new — Новая (до получения ответа 1С)
    • on_review — На рассмотрении (1С ответила, менеджера ещё нет)
    • in_progress — В работе (появился менеджер)
    • in_transit — В пути
    • delivered — Доставлено/завершение этапов
    • closed — Заявка закрыта

    Документы в МП при подаче (начальные, обязательные):

    • passport_front — паспорт, 1 сторона
    • passport_registration — паспорт, 2 сторона
    • inn
    • snils
    • invoice
    • contract
    • payment_check — чек оплаты за авто
    • car_nameplate_photo — фото VIN/таблички авто
    • car_mileage_photo — фото пробега авто
    • car_front_photo — фото авто (перед)
    • car_back_photo — фото авто (зад)

    Дополнительно (по ситуации, не обязательно): add_doc1 и add_doc2.

    Подсказки пользователю: подстатусы 1С используются для того, чтобы в карточке заявки подсвечивать, какие документы нужно подписать/загрузить.

    Подпись — два файла: оригинал (contract) и подпись (contract_sign). При браке подписи — подстатус signature_revision_required.

    Тип сделки: bilateral | cash | tripartite | quadripartite — задаёт менеджер в 1С при статусе «В работе».

    Заявление на перевод: funds_transfer_application — перевод остатков после растаможивания.

    Особые правила: funds_transfer_application, passport_notarized_copy — не из 1С в МП, только *_sign от клиента.

    Для 1СКонтракты и ответственность

    Создание заявки:

    • Сервер инициирует создание заявки в 1С после успешного сохранения в сервере.
    • 1С создаёт карточку заявки со статусом «Черновик».
    • 1С возвращает серверу: { requestId, external1cId }.

    Обновления (1С → сервер): POST /api/integration/customs-requests/state

    • Обязательно: external1cId
    • status, statusSubType, statusSubTypeDateTime
    • dealType — при переводе в работу (см. Справочники)
    • managerExternal1cId, managerFullName
    • Цепочка: new → (ответ создания) on_review → (менеджер) in_progress
    • Возврат подписей: statusSubType: signature_revision_required

    Файлы:

    • На этапе создания сервер передаёт 1С URL исходных файлов из МП (по docType).
    • 1С формирует первичные документы на подпись и квитанции; сервер отдаёт их в МП.
    • После подписи пользователь загружает файлы в МП, сервер пересылает результаты в 1С (через загрузки по docType).

    Справочники: подстатусы и все docType — вкладка Справочники и docs/catalog-reference.md.

    Особые правила по документам:

    • funds_transfer_application (перевод остатков) и passport_notarized_copy — не из 1С в МП, только *_sign из МП.
    • Файлы: upload → URL в state.signingFiles; подстатус от 1С или с сервера.
    • Перезагрузка документов допустима с любой стороны (новая версия по тому же docType).
