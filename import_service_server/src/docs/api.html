API документация

Краткий обзор эндпоинтов приложения и интеграции 1С. Страница подставляет {{BASE_URL}} и WebSocket-базу {{WS_BASE_URL}} из текущего запроса.

Базовый URL API: {{BASE_URL}}

WebSocket (чат): {{WS_BASE_URL}}

Мобильное приложение. Создание: POST /customs-requests без files → N× upload (requestId, uploadIndex, uploadTotal) → на последнем файле create в 1С. Подписи и чеки — тот же upload. Карточка: files[], суммы advancePayment, actualPayment, refundAmount.

    Исходящее создание заявки (HTTP на стороне 1С)

    POSTURL из админкиСервер отправляет заявку после POST /api/customs-requests.

    URL и Bearer задаются в PUT /api/admin/settings/one-c-request-create. Тело — как у приложения + requestId.

    ПолеОбяз.Назначение

    files[]даДокументы при создании: паспорт, ИНН, инвойс, фото авто и т.д. (docType без _sign).

    Исходящий update файлов (HTTP на стороне 1С)

    POSTURL обновления файловСервер передаёт в 1С файлы, загруженные клиентом из МП.

    Auth: тот же Bearer, что для создания (oneCRequestCreateBearerToken в админке).

    URL: отдельное поле oneCRequestUpdateUrl в админке. Если пусто — к URL создания добавляется /files.

    Отдельный роут в 1С: только requestId, external1cId, files[] — файлы после upload из МП. Статусы — через state (1С → сервер).

    Поля тела (сервер → 1С):

    ПолеОбяз.Назначение
    requestIdдаId заявки на сервере.
    external1cIdдаId заявки в 1С.
    files[]даТолько что загруженные из МП файлы (см. ниже).

    Что бывает в files[] из МП (docType):

    • подписи: contract_sign, kuts_sign, …
    • чеки оплаты: payment_recycling_fee_receipt, payment_customs_duty_receipt
    • подтверждения оплаты (фото/скан): payment_recycling_fee, payment_customs_duty
    • доп. документы: add_doc1, add_doc2

    Элемент files[]: docType, fileName, mimeType, fileUrl — 1С скачивает по URL и сохраняет у себя.

    Когда вызывается: после батча POST /api/customs-requests/upload из МП (uploadIndex === uploadTotal). В files[] — только изменённые docType этого батча.

    Request (пример):

    {
  "requestId": 123,
  "external1cId": "GUID-REQUEST-1C",
  "files": [
    {
      "docType": "contract_sign",
      "fileName": "contract_sign.pdf",
      "mimeType": "application/pdf",
      "fileUrl": "https://157-22-173-7.sslip.io/api/customs-requests/files/uuid.pdf"
    }
  ]
}

    Пример с чеком: docType = payment_recycling_fee_receipt — тот же формат элемента files[].

    Исходящая передача сообщений в 1С (HTTP на стороне 1С)

    POST…/customs-request-chatСервер отправляет сообщение пользователя в 1С сразу после POST /api/customs-requests/:id/messages.

    Назначение: как только клиент написал в чате МП, сервер синхронно отправляет это сообщение в 1С по HTTP.

    Auth: тот же Bearer, что и для создания/обновления заявки (из админки).

    URL: отдельный маршрут чата на стороне 1С (не смешивается с созданием заявки и обновлением файлов).

    ПолеОбяз.Назначение
    external1cIdдаId заявки в 1С, к которой относится чат.
    clientMessageIdдаId сообщения на стороне клиента/сервера (дедуп и трассировка).
    textнет*Текст сообщения (*должен быть текст или вложения).
    attachments[]нетВложения: fileUrl, fileName, mimeType.

    Request (сервер → 1С):

    {
  "external1cId": "GUID-REQUEST-1C",
  "clientMessageId": "f14d9f58-f1c0-4f4d-b3ef-f92d0cc7077f",
  "text": "Добрый день, подскажите по сроку выпуска",
  "attachments": [
    {
      "fileUrl": "https://157-22-173-7.sslip.io/api/customs-requests/files/uuid.pdf",
      "fileName": "question.pdf",
      "mimeType": "application/pdf"
    }
  ]
}

    Ожидаемый ответ 1С: любой 2xx и JSON. При наличии id сообщения 1С (oneCMessageId/message1cId/id_1c) сервер сохраняет его в message_1c_id.

    При ошибке 1С: сообщение остаётся в чате МП со статусом доставки failed и текстом ошибки в last_1c_error.

    Общие коды для интеграции: статусы, подстатусы, dealType, docType. Файлы везде — массив files[] (частично, по docType).

    FLOWupload → stateТиповые сценарии.

    1. Пакет документов в МП

    1. N× POST /api/customs-requests/upload — один файл на запрос. Для 1С: Content-Type: application/json, файл в fileBase64 (см. карточку upload ниже). Поля: external1cId, docType, uploadIndex 1…N, uploadTotal=N

    2. Сразу push в МП

    POST/api/customs-requests/uploadФайл в заявку из 1С (JSON + base64) или multipart.

    Auth: Authorization: Bearer <INTEGRATION_BEARER_TOKEN> (1С). МП — Bearer accessToken, только multipart.

    Назначение: прикрепить файл к заявке по docType; повторная загрузка перезаписывает тот же docType.

    Для 1С (рекомендуется): HTTP-запрос с телом JSON. Файл — строка base64 в поле fileBase64. Multipart в 1С обычно недоступен — не используйте multipart/form-data.

    Заголовки (1С, JSON):

    POST {{BASE_URL}}/customs-requests/upload HTTP/1.1
Content-Type: application/json; charset=utf-8

    Тело запроса (1С) — один файл на один POST:

    {
  "external1cId": "GUID-REQUEST-1C",
  "docType": "contract",
  "uploadIndex": 1,
  "uploadTotal": 3,
  "mimeType": "application/pdf",
  "fileName": "contract.pdf",
  "fileBase64": "JVBERi0xLjQK..."
}
    Поле JSONОбяз.Назначение
    external1cIdдаId заявки в 1С.
    docTypeдаКод документа (справочник).
    uploadIndexдаНомер файла в пачке, с 1.
    uploadTotalдаСколько файлов в пачке (для одного файла: 1 и 1).
    fileBase64даСодержимое файла в base64. Альтернативные имена: file, fileContent, base64.
    mimeTypeнетНапример application/pdf, image/jpeg.
    fileNameнетИмя для справки (на сервере имя строится из docType).

    1С: в HTTPЗапрос задайте Content-Type: application/json и тело — сериализованная структура (файл предварительно в base64). Не отправляйте бинарник как multipart.

    Ошибки:

    • 406 FST_INVALID_MULTIPART_CONTENT_TYPE — 1С отправила не JSON и не multipart; для 1С нужен application/json.
    • 403 FORBIDDEN — JSON upload без интеграционного Bearer (только 1С).
    • 400 VALIDATION_ERROR — нет fileBase64, пустой base64, нет external1cId / docType.
    • 401 INVALID_INTEGRATION_TOKEN — неверный Bearer.

    ЗаказчикуЧто происходит с заявкой

    Принцип: МП общается только с сервером. 1С общается только с сервером.

    Этапы (по смыслу документа):

    • 1. Создание заявки — МП: форма → N×upload → сервер создаёт в 1С. Статус «Новая» до ответа 1С.
    • 2. Обработка в 1С — выбор менеджера и подготовка документов.
    • 3. Первичный пакет на подпись — 1С → МП, затем подпись и возврат в 1С через сервер.
    • 4. Квитанция утилизационного сбора (параллельно п.3) — 1С → МП, затем загрузка чека.
    • 5. Квитанция госпошлины (параллельно п.3) — 1С → МП, затем загрузка чека.
    • 6. Архив перед транзитом — 1С → МП (только скачивание).
    • 7. Итоговые документы после таможни — 1С → МП (только скачивание).

    Статусы (отображение в МП):

    • new — Новая (до получения ответа 1С)
    • on_review — На рассмотрении (1С ответила, менеджера ещё нет)
    • in_progress — В работе (появился менеджер)
    • in_transit — В пути
    • delivered — Доставлено/завершение этапов
    • closed — Заявка закрыта

    Документы в МП при подаче (начальные, обязательные):

    • passport_front — паспорт, 1 сторона
    • passport_registration — паспорт, 2 сторона
    • inn
    • snils
    • invoice
    • contract
    • payment_check — чек оплаты за авто
    • car_nameplate_photo — фото VIN/таблички авто
    • car_mileage_photo — фото пробега авто
    • car_front_photo — фото авто (перед)
    • car_back_photo — фото авто (зад)

    Дополнительно (по ситуации, не обязательно): add_doc1 и add_doc2.

    Подсказки пользователю: подстатусы 1С используются для того, чтобы в карточке заявки подсвечивать, какие документы нужно подписать/загрузить.

    Подпись — два файла: оригинал (contract) и подпись (contract_sign). При браке подписи — подстатус signature_revision_required.

    Тип сделки: bilateral | cash | tripartite | quadripartite — задаёт менеджер в 1С при статусе «В работе».

    Суммы таможни: 1С передаёт advancePayment (аванс) и actualPayment (факт на день таможни) в state; сервер отдаёт в МП три суммы, включая refundAmount (аванс − факт).

    Заявление на перевод: funds_transfer_application — перевод остатков после растаможивания.

    Особые правила: funds_transfer_application, passport_notarized_copy — не из 1С в МП, только *_sign от клиента.

    Для 1СКонтракты и ответственность

    Создание заявки:

    • Сервер инициирует создание заявки в 1С после успешного сохранения в сервере.
    • 1С создаёт карточку заявки со статусом «Черновик».
    • 1С возвращает серверу: { requestId, external1cId }.

    1С → сервер: upload (файлы) + state (метаданные) — вкладка API для 1С, docs/contract-files-v2.md.

    Сервер → 1С (отдельный URL обновления файлов): только requestId, external1cId, files[] — подписи, чеки и т.д. из МП. URL задаётся в админке (oneCRequestUpdateUrl) или как create + /files.

    Создание заявки (МП): POST /customs-requests без файлов → N×upload → create в 1С на последнем файле.

    Создание (сервер → 1С): после upload из МП — requestId + files[] с URL.

    Справочники: подстатусы и все docType — вкладка Справочники и docs/catalog-reference.md.

    Особые правила по документам:

    • funds_transfer_application (перевод остатков) и passport_notarized_copy — не из 1С в МП, только *_sign из МП.
    • Файлы от 1С — upload; от МП в 1С — отдельный URL (oneCRequestUpdateUrl или create + /files).
    • Перезагрузка документов допустима с любой стороны (новая версия по тому же docType).
